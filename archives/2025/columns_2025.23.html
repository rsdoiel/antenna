A CNAME for your feed

date: 2025-06-10, from: Dave Winer‚Äôs Scripting News

Here‚Äôs something to think about. As you may recall, I have an hourly art feed, available both in RSS and on Bluesky. Same content.

Today I gave each the same new name, art.feediverse.org.

If you use that URL on Bluesky, you go to its version of the content stream. And if you give the same URL to a web browser, ie accessing it via HTTP, you get the RSS version of the same data.

  • On Bluesky: https://bsky.app/profile/art.feediverse.org.
  • On the web: https://art.feediverse.org.

BTW ‚Äì think of feediverse as a web of feeds. It‚Äôs open of course, because the web itself is open. And it‚Äôs pretty social as you will see. It‚Äôs why I think it‚Äôs really the first instance of the ‚Äúopen social web.‚Äù

I‚Äôve long felt documents should have their own CNAMEs. I even went so far as to design a system where each node in an outline could have a CNAME. I still believe in this, why not have a hierarchic file system that‚Äôs as editable as an outline? It‚Äôs the same idea that works in so many other contexts. It‚Äôs why the idea of a scripting system on a graphic PC like the early Mac made a world of sense. It‚Äôs why Unix underneath that same UI was the right way to go. And why the web is so nice, you just View Source and you get to lift the hood and see how it works.

They were smart at Bluesky to use DNS this way. Why invent your own identity system when the net itself has a great distributed system that scales? They just made the wrong call (imho) in not using RSS for their timeline data, or at least the outside world‚Äôs view of their timeline, in both directions ‚Äì in and out. That‚Äôs the basic idea of the Internet itself. No one cares what‚Äôs behind your TCP interface, it could be a network with a million nodes using some wacky protocol to connect them. As long as I can get to you via TCP, you‚Äôre on the net. This is the kumbaya philosophy of the net, why not just use what already works.

You‚Äôll see that we‚Äôre able to do some interesting twitter-like stuff with this duality.

2025-05-27 Alt text on this blog

On the web, the alt text is a description of an image to be shown by browsers when the image cannot be shown ‚Äì like text browsers or screen readers. So what about alt text on this blog? What about blog posts full of photos, specifically? Like the Croatia posts.

I think that alt text should be unnecessary on my blog because the text should carry the entire message. ‚ÄúWelcome to my Wall of Text!‚Äù üòÑ

On fedi, I like alt text and I both read and write it. I hate clients that don‚Äôt display it. I want to always read it. Therefore, on my blog, the equivalent of alt text should always be readable. But real alt text isn‚Äôt shown to me ‚Äì the browser is showing me the pictures, instead. It makes sense. And on fedi it only works because the applications and front-ends don‚Äôt just use the alt text attribute for images in HTML ‚Äì they have an extra user-interface to allow me to read it. Buttons, mouse-over, long-press, whatever it takes.

There are options for this blog, of course. Except I don‚Äôt like these options.

I could reuse the alt text as the image title attribute which is shown on mouse-over by default. I don‚Äôt like being forced to use mouse and I don‚Äôt like duplicating the description.

In Markdown: 

![alt text](url "title")

I could automatically use the alt text as the title attribute but I still don‚Äôt like to use the mouse to read the title attribute and I wouldn‚Äôt be able to switch Markdown rendering engines as easily if I hack it. Writing some Javascript to produce extra pop-ups or some other user-interface element would be even worse, and it would need extra clicking or keyboard navigation as a reader. No way.

The last option is the best. Just write the description above the image: 

Description

![](url)

I‚Äôve tried using the title attribute for some of my recent picture-heavy blog posts about Croatia and I‚Äôve been checking the result using Gemini and Gopher, and I‚Äôm not happy.

Compare the following two with the web version where I can‚Äôt read the alt text. (Screenshots from Lagrange on my phone.)

The Gemini version looks OK with some text paragraphs being links to the images.

The Gopher version shows the alt texts as weird disjointed text paragraphs that sometimes repeat the preceding paragraph and sometimes don‚Äôt.

I guess I could write longer alt texts until the Gopher version looks good but then the Gemini version looks weird and the web version continues to not show it.

Now check the Gemini and Gopher output for the page you‚Äôre reading right now!

The Gopher version is OK. The line-breaks don‚Äôt look nice on the phone at the default resolution so ignore that weird line-break. What we can see is that the empty alt text generated an empty paragraph. Something I should fix. üòÖ But in general, it looks OK and reads well.

The Gemini version still looks OK. It also links to the Gopher and Gemini versions, so ignore those extra links. üòÖ The browser substitutes the filenames for the missing alt text. That works for me. I‚Äôm not sure what a screen reader would do. Probably say: ‚ÄúImage. 2025-05-27-alt-text-3.jpg‚Äù? And I‚Äôm not sure what a person with bad sight would prefer. At this point I‚Äôd know that I can skip the image since I have already read the description above.

And here‚Äôs the same passage a day later, using the text browser lynx. The inline links look good, and there‚Äôs no trace of the screenshots. This is great! It‚Äôs as readable as the Gopher version with no extra line-breaks, and the inline links are inline (unlike in the Gemini version).

Checking with edbrowse. The inline links are wrapped in bracers, which looks good, and the images with no alt text are shown as empty square brackets ([]) which works for me. Ideally, I guess, they‚Äôd just disappear.

All in all, I think this solution is better: Instead of alt text, just write more text above each image.

What I‚Äôm less sure about is whether this automatically implies that all images are ‚Äúdecorative‚Äù and therefore should have an empty alt text (instead of no alt text). It seems to me that the filename being shown is a lot of visual clutter, but that only seems to affect Gemini, not the text browser.

What do you think?

2025-05-31. I added a piece of Javascript to copy the alt text to the title attribute if the alt text is set and the title is not. Perhaps that‚Äôll motivate me to add more alt text.

<script type="text/javascript">
function addTitle() {

  for (let img of images) {

2025-06-09. Tweaked the JavaScript. I‚Äôm now replacing the img element with a figure element, moving the img element into the figure and adding a figcaption element. The caption is rendered smaller and italic via CSS.

function addTitle() {

  for (let img of images) {
    const parent = img.parentNode;
    const figure = document.createElement("figure");

  }
}
